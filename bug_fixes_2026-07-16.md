# Near & Now — Full Codebase Deep Audit (2026-07-16)

Full-codebase pass across all 7 areas: `backend` (Express API), `frontend` (customer website), `admin` (admin panel), Supabase schema/migrations, and all three mobile apps (`near-now-store_owner`, `NAT_Near-Now_Rider-`, `nearandnowcustomerapp`). This supersedes/extends `bug_fixes_2026-07-15.md` (admin-approval-flow focused) and `supabase/SCHEMA_NOTES.md` (schema-focused) — read those first for what was already fixed on 2026-07-15/18.

**Nothing in this file has been fixed yet.** This is the research pass only — findings are listed, not yet applied. Given the volume and severity (several live, exploitable, financial/security-critical bugs), fixing everything needs prioritization, not a blind autofix pass.

---

## 🔴 TOP PRIORITY — fix these first, in roughly this order

1. **`/api/payment/refund` has zero authentication** (backend, Critical) — anyone can refund any real order given just a Razorpay payment ID.
2. **Any signed-up delivery partner can hijack/complete/reset ANY order** (backend, Critical) — `acceptOrder`/`rejectOrder`/`markPickedUp`/`markDelivered` have no ownership check tying the order to the calling rider.
3. **Checkout order-total/price/payment-status is fully client-trusted** (backend `placeCheckoutOrder`, Critical) — a client can set `order_total: 1, payment_status: "paid"` and the order is created as paid.
4. **Store-owner and delivery-partner signup never verifies OTP actually happened** (backend, Critical) — `POST .../signup/complete` trusts the phone number in the body with no proof an OTP was verified; full account-creation auth bypass, scriptable.
5. **Customer app: Supabase service-role key is structurally required client-side** (customer app, Critical) — core features (order placement, addresses, login) call `assertSupabaseAdminConfigured()`; either the service-role key ships inside every APK (full DB compromise) or these features are broken in production. Needs an architecture fix (route through backend), not a config tweak.
6. **Customer app: full IDOR across profile/orders/addresses** (customer app, Critical) — every Supabase-direct call takes a bare `userId` string with zero verification it matches the authenticated session.
7. **Tracking endpoints leak any order's address/driver PII to any logged-in customer** (backend, High) — `getOrderTracking*` never checks the order belongs to the caller.
8. **Admin login still does client-side `bcrypt.compare()` reading `admins.password_hash` via anon-readable RLS** (admin, Critical — previously known, confirmed still live) — the "secure" Edge Function login path exists but is dead code; the real login flow is the vulnerable one.
9. **Delivery/pickup OTP and pickup codes are brute-forceable** (backend, High) — 4-digit codes, no rate limiting, directly gate "proof of delivery."

> ~~Website double-charges GST on every order~~ — **not a bug.** Confirmed by product owner: the flat 5% added at checkout on top of GST-inclusive product prices is intentional by-design pricing, not double-taxation. See "Confirmed NOT a bug" note under Website frontend below.

Everything else below is organized by area, severity within each area.

## Backend (`backend/src`)

### Critical

- **No ownership check on order-mutating rider endpoints.** `deliveryPartner.controller.ts`: `acceptOrder()` (398-435), `rejectOrder()`'s `customer_orders` update (450-453), `markPickedUp()` (468-493), `markDelivered()` (495-532) — all update by `orderId` alone, no check the order is assigned to `req.riderId`. Any authenticated rider can call `POST /delivery-partner/orders/<any-id>/delivered` and falsely mark a stranger's order delivered (triggers real customer notifications), or steal/reset an order actively being handled by someone else.
- **`requireRider` never checks `is_approved`.** `deliveryPartner.controller.ts:28-58` — only validates the session token exists/isn't expired. `acceptOrder`/`acceptOffer` individually check `is_approved`, but `rejectOrder`/`markPickedUp`/`markDelivered` don't — so a `pending_verification` rider signed up seconds ago can still call those three.
- **`/api/payment/refund`, `/methods`, `/:paymentId`, `/create` have zero authentication.** `payment.routes.ts` (whole file). `processRefund` — anyone with a `paymentId` can trigger a real refund. `getSavedMethods` — `GET /api/payment/methods?user_id=<any-id>` discloses another customer's saved card/UPI details.
- **`placeCheckoutOrder` trusts client-supplied prices/total/payment_status.** `orders.routes.ts:68` (unauthenticated, acknowledged as SECURITY-010 in a code comment) → `database.service.ts:722-1034`. `unit_price: item.price`, `total_amount: orderData.order_total`, `payment_status: orderData.payment_status` all come straight from the request body with no server-side recomputation from the `products` table.
- **`signupComplete` (store owner + delivery partner) never verifies OTP was actually completed.** `storeOwner.controller.ts:214-312`, `deliveryPartner.controller.ts:91-135`, both unauthenticated routes. No token/flag proves the phone passed `/api/auth/verify-otp` first — trusts the `phone` field outright. Scriptable mass-registration of fake accounts.

### High

- **`storeOwner.routes.ts` has no router-level auth middleware**, unlike every other role's router. `deleteStoreProduct` (103-151) looks up `app_users` by session token without filtering `role='shopkeeper'`, unlike the other handlers in the same file — inconsistent auth pattern, latent IDOR if `stores.owner_id` is ever set programmatically for a non-shopkeeper.
- **`cancelOrder` has no order-status guard.** `database.service.ts:419-506` only blocks cancellation if a delivery partner is assigned — never checks if the order is already `order_delivered`. A delivered order can be "cancelled" and auto-refunded (`payment_status === 'paid'` triggers `processRefund`).
- **`updateOrderStatus` (admin) has no state-machine validation.** `orders.controller.ts:205-265` — any admin session can move `order_delivered` back to `pending_at_store`, re-triggering customer notifications inappropriately.
- **Tracking endpoints leak PII cross-customer.** `tracking.controller.ts` — `getOrderTracking`, `getOrderTrackingFull`, `getDriverLocationsForOrder`, `getTrackingHistory` never check the requested `orderId` belongs to `req.customerId`, despite being `requireCustomer`-gated. Any logged-in customer can view any other customer's delivery address + rider phone/location by guessing/observing an order UUID.
- **Riders can spoof another rider's GPS / write arbitrary order status.** `tracking.controller.ts:107-122` (`updateAgentLocation`) never checks `agentId === req.riderId`. `addTrackingUpdate` (64-87) accepts any status string with no ownership check and no enum validation, writing directly into `customer_orders.status` — despite a code comment explicitly saying this route "must not be open."
- **Delivery/pickup OTP and pickup codes are brute-forceable.** `verifyDeliveryOTP` (537-567), `verifyPickupCode` (862-934) — 4-digit codes, no rate limiting, no lockout.

### Medium

- **Coupon `usage_count` increment is non-atomic (read-then-write race).** `database.service.ts:1199-1215`, acknowledged in its own comment as "unlikely but not impossible" — concurrent checkouts can both pass the usage-limit check before either write lands, letting a `usage_limit: 1` coupon be redeemed multiple times under burst traffic.
- **Google Places/Geocoding/Directions proxy is fully public with no rate limiting.** `places.routes.ts` (all 6 routes) — unlike the OTP endpoints, nothing throttles this; scriptable to run up Google Maps billing (cost-DoS).
- **`getOrderById` returns 500 instead of 404** for a non-existent order (`.single()` throws before the `!order` check can run) — `database.service.ts:402-417` / `orders.controller.ts:188-203`. Worth a broader audit of `.single()` vs `.maybeSingle()` usage.
- **Admin-facing coupon/delivery-partner endpoints have no schema validation** (unlike `orders.routes.ts`, which uses `validate(schema)`) — `coupons.controller.ts:29-49`, `delivery.controller.ts:50-75` pass `req.body` straight to the DB layer.
- **`notifications.controller.ts` has the same IDOR *pattern*** as the tracking bugs (no `req.customerId` cross-check on `userId`/`notificationId`), but is currently harmless because the backing service (`database.service.ts:1668-1687`) is a complete stub (always returns `[]`, no-op writes). Will become exploitable the moment someone implements it without also adding the ownership check the route comment incorrectly claims already exists.

### Low

- **`auth.controller.ts:195,337`** calls `crypto.randomUUID()` without importing `crypto` — works only because Node's global `crypto` happens to be available unprefixed; fragile across environments/bundlers.
- **`notification.service.ts:113-117`** `sendSMS` is a dead stub (`// TODO: Twilio / AWS SNS`) that always returns success without sending anything.
- **`requireShopkeeper` blocks even read-only profile access** for unapproved shopkeepers (`shopkeeper.controller.ts:30-71`) — a newly-signed-up shopkeeper can't check their own approval status.
- **Expo push send calls never inspect per-message error responses** — `delivery.controller.ts:232-241`, `shopkeeper.controller.ts:499-511,550-560` — stale/invalid tokens never get cleaned up, failures are invisible.
- **Duplicate `declare module 'express'` Request augmentation** in `invoice.controller.ts:9-15`, already redundant with 3 other files — the file's own comments document this exact duplication caused a real bug once before.

## Website frontend (`frontend/src`)

### Confirmed NOT a bug

- **Flat 5% added at checkout on top of GST-inclusive product prices** (`utils/checkoutCalculations.ts:88`, `pages/CheckoutPage.tsx:426-427,603-604`) — flagged in the initial pass as a "double GST charge." **Confirmed by product owner: this is intentional by-design pricing, not a bug.** No code change made.

### Critical

- **Client fully computes the amount sent to Razorpay, with no visible server-side recheck.** `pages/CheckoutPage.tsx:427` → `services/paymentGateway.ts:76-89` posts a client-computed `amount` to `/api/payment/create`. Combined with the backend's `placeCheckoutOrder` trusting client-supplied totals (see Backend Critical above), there's no verified guardrail preventing a manipulated total from reaching payment creation.
- **Direct-from-browser order-write fallback path.** `utils/apiBase.ts:18-21` (`shouldUseBackendApi`) + `services/supabase.ts:483-750` (`createOrder`) — if `VITE_API_URL` is ever unset in a production build, the app falls back to writing `customer_orders`/`store_orders`/`order_items` directly from the browser via an anon-key client, letting the client set `order_total`/`discount_amount`/`payment_status` directly, relying entirely on unverified RLS to prevent abuse.

### High

- **Logout doesn't clear the cart.** `context/AuthContext.tsx:166-186` (`logoutUser`) never calls `clearCart()` — confirmed via grep that no logout call site does. Shared/kiosk device: User B logs in after User A logs out and sees/can check out User A's cart. `LocationContext`'s saved location is likewise never cleared.
- **No global 401 handling on the customer-facing site.** Only `ShopkeeperApp.tsx`/`DriverApp.tsx` (separate legacy pages?) check `status === 401`; the actual customer flow (`authService.ts`, `orderService.ts`, `supabase.ts` backend-API branches) never detects an expired/revoked session, clears it, and redirects to login — user gets stuck with `isAuthenticated: true` but every API call silently failing.
- **`useOrderTrackingRealtime.ts:64-96,99-147` — stale closure bug.** The realtime-subscription effect's dependency array is `[orderId, !!order]` (a boolean), not `[orderId, order]`, so the closure keeps referencing the *first* render's `order` object for the subscription's lifetime. Any order field not explicitly re-assigned in each update permanently reverts to its first-render value and never updates again.

### Medium

- **`NotificationContext.tsx:34-48`** — auto-dismiss timers reset for *all* visible toasts every time the notifications array changes (add or remove), so bursts of notifications within the dismiss window can pile up or overstay far longer than configured.
- **`HomePage.tsx`** never re-fetches products when the user's delivery location changes (unlike `ShopPage.tsx`, which correctly does) — stale "nearby store" listing until a full reload.
- **`SearchPage.tsx:15-39`** — no request cancellation/staleness guard; rapid searches can have an older, slower response overwrite a newer search's results.
- **`ThankYouPage.tsx:10-17,131-158`** — order details come only from React Router `location.state` with no fallback fetch by order ID; refreshing the post-payment Thank You page (very common — checking email, app-switching) loses all context and shows "Order ID: N/A".
- **Two parallel, inconsistent admin-auth systems** in the same codebase — `secureAdminAuth.ts`'s Edge-Function JWT login is referenced by several components but the real login flow (`AdminLoginPage.tsx`) uses a completely different mechanism (`adminAuthService.ts`), meaning the "secure" path is dead code that's easy to mistakenly believe is what's actually running (this is the same root confusion behind the still-live client-side-bcrypt admin login bug in the Top Priority list).
- **API-base-URL resolution logic duplicated in 3 places** (`utils/apiBase.ts`, `services/authService.ts:44-52`, `services/orderService.ts:4-12`) instead of sharing one implementation — drift risk.
- **`utils/encryption.ts`** — dead code, unused anywhere; client-side "encryption" with a bundle-embedded key (defaulting to a hardcoded weak string if unset) provides no real confidentiality even if it were wired up.

### Low

- Greedy store-assignment algorithm in `createOrder`'s fallback path uses `items.indexOf()` inside nested loops — effectively O(n³) for cart size n (harmless at typical cart sizes, but wasteful).
- 60+ files contain unstripped `console.log`/`console.error` in production, including full order payloads and user IDs/phones.
- `utils/dataMasking.ts` — confirmed dead code, zero importers.
- Security utilities (`sanitize.ts`, `csrf.ts`, `rateLimit.ts`) are applied only to the admin surface, never to customer-facing forms — inconsistent application, though no direct injection sink was found on the customer side.

## Admin panel (`admin/src`)

### Critical

- **Client-side login still reads `admins.password_hash` via an anon-readable RLS policy.** `services/adminAuthService.ts:85-104` (`authenticateAdmin`, the function actually wired to the login form) calls `getAdminClient()` *before* any session exists, which falls back to the plain anon-key client — then does `bcrypt.compare()` **in the browser** against a hash fetched over the public anon key. Anyone can call the Supabase REST endpoint directly with the public anon key and dump every admin's password hash for offline cracking, if the `admins` RLS policy still permits anon SELECT (matches a bug already flagged in `bug_fixes_2026-07-15.md` as known-but-unfixed — confirmed still live here).
- **`getAdminClient()` silently downgrades to the anonymous client on any missing/expired token — for every call site, not just login.** `services/supabase.ts:46-55`. Every function in `adminAuthService.ts`, `adminService.ts`, `auditLog.ts`, and every page calling `getAdminClient()` directly runs as fully anonymous the moment the session token is gone, with no error thrown — just silently executes against whatever RLS allows `anon` to do. Given the `admins` table's RLS gap above, this is a wider exposure than just login.

### High

- **Same unchecked-mutation bug class as the already-fixed `StoresPage.toggleApproval`, still present in `NotificationsPage.tsx` and `AdminHeader.tsx`.** `NotificationsPage.tsx:221-239` (`markAllRead`/`markOneRead`/`deleteNotification`) and `:76-82` (`PushNotificationPanel.handleSend`), plus `AdminHeader.tsx:118-120` (`markAllRead`) — all call `.update()`/`.delete()`/`.insert()` against `admin_notifications` with **no error or row-count check**, then optimistically update local state regardless of whether the DB call actually succeeded. A stale/expired session makes these silently no-op while the UI reports success.

### Medium

- **Session-creation writes in `authenticateAdmin` are unchecked.** `adminAuthService.ts:112-127` — the `admin_sessions` insert and `last_login_at` update results are discarded entirely (not even logged). If the session insert silently fails, the UI hands back a token that *looks* valid but has no corresponding DB row — every subsequent `is_admin_authenticated()` check fails, and the admin appears logged in while every real page shows empty data (this is precisely the failure mode already root-caused and fixed once on 2026-07-15 for the *first* login after that fix — this finding is about the same class of bug being able to recur silently for any future login where the insert happens to fail).
- **`adminService.ts:802-811` (`updateCustomerStatus`) is a dead no-op** — takes unused params, logs a warning, returns `null` without touching the DB. Currently unused by any button, but a trap if resurrected.
- **`AdminSidebar.tsx`'s `isActive('/')` check references a `/admin` path that doesn't exist** anywhere in the app's actual routing (mounted at `/`) — the Dashboard nav item can never show as active. Cosmetic, but the kind of leftover-after-refactor bug worth cleaning up.

### Low

- Client-side-only rate limiting (`utils/rateLimit.ts`) — an in-memory Map reset by any page refresh, providing no real protection for the vulnerable login flow above.
- Admin email/role/error-stack details logged to console on every login attempt, uncompressed for production.
- `services/supabase.ts` inside the *admin* app contains a large amount of customer-facing logic (`createOrder`, `getUserOrders`, newsletter, address CRUD) that's dead weight in the admin bundle — looks like a stale copy-paste from the customer-facing frontend.

### Confirmed NOT bugs (verified during this pass)

- All backend-API `fetch()` calls needing admin auth correctly attach the `Authorization` header now.
- All `.from()` calls in pages/services (aside from the pre-login fallback above) correctly use `getAdminClient()`.
- `NotificationProvider` and `ErrorBoundary` are both correctly mounted (fixed earlier today).

## Supabase schema & migrations

### Critical

- **`supabase/migrations/` cannot bootstrap this schema from an empty database — this is systemic, not isolated to `admin_sessions`.** Grepping every `CREATE TABLE` for `coupons`, `app_users`, `stores`, `admins`, `categories`, `master_products`, `order_items`, `store_orders`, `coupon_redemptions`, `product_images`, `product_attributes`, `product_details`, `product_reviews`, `audit_logs`, `security_events`, `failed_login_attempts`, `customers`, `driver_locations`, `admin_activity_logs`, `inventory_logs` across every migration file returns **zero matches for all of them** — only `delivery_partners` has a real tracked `CREATE TABLE`. `supabase/SCHEMA_NOTES.md` currently frames the untracked-table problem as isolated to `admin_sessions`; it should be corrected to say this is the norm for this repo, not an exception. Practical impact: `supabase db reset` or spinning up a fresh project from these migrations alone would fail immediately.
- **`customer_orders` (`20260404120000_customer_orders_razorpay_payment_id.sql:109`) declares its `payment_method` column as type `public.payment_method_type` — a type that is never `CREATE TYPE`'d anywhere.** The enum actually created later (`20260405000000_..._enums_payouts.sql:38`) is named `public.payment_method` (no `_type` suffix). A later migration even has defensive code detecting a mismatched `udt_name`, implicitly acknowledging this — but that defensive code only works if `payment_method_type` already exists out-of-band in the live DB. Replayed purely from tracked migrations, this fails with `type "public.payment_method_type" does not exist`.
- **`public.order_status` — the single most important enum in the app (drives all order lifecycle transitions) — has no `CREATE TYPE` anywhere in migrations either.** Used from `20260404120000` onward and later `ALTER TYPE ... ADD VALUE`'d, but never created.

### High

- **Every `SECURITY DEFINER` function in the migration history is missing `SET search_path = public`.** Full list: `accept_driver_offer()`, `auto_offline_stale_drivers()`, `is_account_locked()`, `is_admin_authenticated()` (re-created on 2026-07-17 still without the fix), `notify_admin_new_order()`, `notify_admin_new_user()`, `debug_admin_headers()`. `is_admin_authenticated()` is the most sensitive — it's the sole gate for every `admin_full_access` RLS policy across 9 tables. This is a known, standard Postgres privilege-escalation hardening gap.
- **`debug_admin_headers()` — the temporary diagnostic function from the 2026-07-17 investigation — was never dropped**, despite its own comment saying "safe to drop once the admin-auth 0-rows issue is resolved" (which it has been, as of the very next day's migrations). It's `SECURITY DEFINER`, has no `search_path` pin, is granted to `anon`/`authenticated`, and returns the full matching `admin_sessions` row (including `session_token`) to any caller who can supply a valid `x-admin-token`. Should be dropped or restricted to `service_role` only.

### Low / informational

- The `coupons`/`product_images`/`product_attributes`/`product_details` grants added on 2026-07-18 (`SELECT` to `anon`/`authenticated` while RLS stays disabled on those tables) is exactly the trap `SCHEMA_NOTES.md` itself warns about — appears deliberate and carefully scoped (non-PII columns only, `product_reviews` explicitly excluded for its PII columns), but worth a second confirmation pass that RLS staying disabled there is intentional, and updating SCHEMA_NOTES to note the trap has since been (carefully) sprung rather than just theoretical.
- Three RLS policies (`customer_own_orders`, `customer_own_record`, `partner_own_record`) are permanently dead code — they gate on `auth.uid()`, but this app doesn't use Supabase Auth (uses its own phone-OTP JWT), so `auth.uid()` is always NULL from the frontend, documented in an unrelated migration's own comment. Not exploitable (fails closed), but misleading — a future developer could believe there's a working customer-self-access RLS path when there isn't.

## Store-owner app (`near-now-store_owner`)

### Critical

- **Push notification service is never initialized at app startup — notifications are effectively opt-in-and-hidden, not opt-out.** `lib/notifications.ts`'s `notificationService.initialize()` has no caller anywhere in the app (confirmed via full-repo grep) — the only way a token gets registered is if the shopkeeper manually opens Settings → Notifications → taps "Enable," despite the in-app copy implying it's already on by default. A shopkeeper who never visits that screen will never receive a push for new orders, silently missing them if they're not actively watching the app.
- **No global 401 handling — same bug class already flagged (and left unfixed) in the customer app.** `lib/api-client.ts`'s `request()` never inspects 401s to trigger logout; `error-handler.ts` has a `handleAuthError()` that's dead code (never called). Every screen just shows empty data indefinitely on an expired/revoked session, with no path back to login short of manually finding Settings → Logout.

### High

- **`clearSession()` doesn't clear the store-list cache — real cross-account data leak window on shared devices.** `session.ts:56-63` never calls `clearStoreCache()` from `lib/appCache.ts`, which owns a 10-minute in-memory + AsyncStorage cache. If shopkeeper B logs in within 10 minutes of shopkeeper A logging out on the same device, `home.tsx`/`previous-orders.tsx`/`payments.tsx` will briefly render **A's** store name/address/approval status before B's real fetch overwrites it.
- **Push token is never cleared on logout or re-associated on a different user's login** — same root cause as the initialize() gap above; a shared device could keep delivering one shopkeeper's order notifications to whoever's currently logged in.
- **`useRequireStoreApproval()` is reactive, not preventive.** The redirect only fires after an async network round-trip resolves; meanwhile the Orders/Payouts/Inventory screens' own data-fetching effects run in parallel, unconditioned on the approval check — an unapproved shopkeeper can view/accept/reject real orders and edit inventory for the duration of that request, and if the request itself errors out, the guard fails open **silently forever** (never redirects at all).
- **`app/inventory.tsx` — an orphaned legacy route with no approval guard at all**, still deep-link-reachable via Expo Router's automatic file-based routing even though no in-app navigation targets it; writes directly to the DB via `lib/storeProducts.ts`.

### Medium

- Realtime channel names include `Date.now()` (`home.tsx:216,223`), causing churn on every store-selection/re-render instead of reusing/deduping channels — confirmed still present from a prior audit.
- Client-side "auto-complete order after 2 minutes" logic (`previous-orders.tsx:285-305`) only runs while the app happens to be open and polling — if the shopkeeper backgrounds/closes the app, orders can be stuck un-completed indefinitely; this should be a server-side scheduled job, not client-triggered.
- Two entirely dead-code order-management implementations exist in parallel with the real one: `hooks/useOrders.ts` + `components/IncomingOrderPopup.tsx` (a more complete 60s-auto-reject flow, never wired in) and `lib/order-service.ts` + `components/OrderFilters.tsx` (targets a `/store-owner/orders/*` API contract that doesn't match the real `/shopkeeper/allocations/*` endpoints in use — would silently fail if ever revived).
- `context/AppContext.tsx` and `lib/testSupabase.ts` (including a function that performs live inserts/deletes against the real `products` table) are both entirely dead/unreferenced code.
- `CachedStore` type doesn't declare `is_approved`, so all the approval-gate logic throughout the app relies on untyped `as any` casts — a backend field rename would fail silently with no compiler error.

### Low

- Several silent `catch {}` blocks in approval-check/session flows (already partly known/intentional, re-confirmed present).
- Heavy unconditional `console.log`/`warn`/`error` in `lib/notifications.ts`, not gated behind `__DEV__`, including full notification payload contents.
- `scheduleNotification()`'s `_triggerDate` parameter is entirely ignored — always fires immediately regardless of what's passed, misleading if a "daily summary" feature (listed as a settings option with no backing implementation) is ever built on top of it.

## Rider app (`NAT_Near-Now_Rider-`)

### Critical

- **Supabase service-role key is plumbed into the client bundle via `app.config.js`.** `constants/config.ts:33-37` + `app.config.js:86-90` — if `EXPO_PUBLIC_SUPABASE_SERVICE_ROLE_KEY` is ever set at build time (an easy mistake alongside the anon key), the full-privilege key ships inside every installed APK/IPA, extractable via decompilation. Nothing in the app currently *uses* it, but the plumbing that would ship it exists.

### High

- **Unauthenticated fallback leaks every rider's PII.** `app/(tabs)/profile.tsx:110-135` — if `/delivery-partner/profile` doesn't return success, falls back to fetching the *entire* delivery-partner roster (`/api/delivery/partners`) with no token, exposing every other rider's verification documents, phone, and address just to client-side `.find()` the current user's own record.
- **App requests "Always" location permission but never actually does background tracking.** `app.config.js` configures `NSLocationAlwaysUsageDescription` and the always-permission plugin, but the code only ever calls `requestForegroundPermissionsAsync()` and uses foreground-only `watchPositionAsync` + a JS `setInterval` heartbeat — both of which stop firing when the app is backgrounded/locked. A rider mid-delivery with a locked phone has stale location in `driver_locations` while still being dispatched new offers against that stale position. Also a plausible App Store review risk (declaring an unused permission purpose).
- **Reinstatement from suspended/offboarded is invisible until app restart.** The status-approval poll only starts when `driverStatus === "pending_verification"` — a rider who was `suspended` when the app loaded, then reinstated by an admin while the app stays open, has no mechanism to discover the change short of force-quitting and relaunching.

### Medium

- Double-tap on "Accept offer" can fire two concurrent accept requests before React's `disabled` state re-render lands — no `useRef` in-flight lock, only async state (server-side atomicity likely prevents double-award, but produces duplicate network calls/alerts/navigations).
- A realtime-channel subscribe/teardown race (`home.tsx:321-356`) can orphan a Supabase channel subscription if `isOnline` toggles quickly between the effect running and its async session-fetch resolving — leads to duplicated haptic buzzes/fetches for new offers over a session.
- Rider-facing "Earnings" screen hardcodes a 15% commission client-side in two separate places (`profile.tsx:147`, `earnings.tsx:90-98,163,217`) with no backend-provided actual payout field — will silently drift from real payouts if the commission structure ever varies by store/promo or changes.
- "Ignore offer" is client-state-only, unpersisted — resets on any app restart, so a previously-ignored offer can reappear.

### Low

- `lib/authService.ts` is entirely dead code (never imported), and its unused OTP functions log raw phone numbers and API URLs — latent PII-in-logs risk if ever wired back in.
- `components/MapViewWrapper.tsx` is unused dead weight — the app bundles `react-native-maps` and Google Maps keys for no actual in-app map usage (only deep-links out to the native Maps app).
- Same "state instead of ref for in-flight-request guard" pattern recurs in `otp.tsx`'s verify flow as in the accept-offer bug above — low-likelihood double-submit race.
- `isActivated` session field continues to be threaded through every session-save call site despite being confirmed dead/unused elsewhere — easy to mistake for the real approval gate.

## Customer app (`nearandnowcustomerapp`)

### Status of 6 previously-known issues: all 6 confirmed still present, unfixed
(logout doesn't clear cart/push token; `apiClient.ts` 401 handling doesn't clear session; `trackingService.ts` treats any failure as "Order not found"; cold-start push deep links can be dropped; no push-token unregister on logout; dead "Order Placed!" overlay in checkout.tsx)

### Critical (new)

- **The Supabase service-role key is structurally required client-side for core app function, not just a misconfiguration risk.** `lib/supabase.ts:31-54` builds an admin client from `SUPABASE_SERVICE_ROLE_KEY || EXPO_PUBLIC_SUPABASE_SERVICE_ROLE_KEY || SUPABASE_ANON_KEY`. Since only `EXPO_PUBLIC_*` vars get inlined into the RN bundle, the non-public `SUPABASE_SERVICE_ROLE_KEY` is `undefined` at runtime on a real device — meaning in production, **either** `EXPO_PUBLIC_SUPABASE_SERVICE_ROLE_KEY` must be set (shipping the full-access key inside every installed app, trivially extractable, giving anyone unrestricted read/write to every table) **or** `createOrder`, all address CRUD, login/profile fetch, and coupon-viewing are simply broken. This is called from `orderService.ts`, `addressService.ts`, `authService.ts`, `product/coupons.tsx`, `distanceUtils.ts`. This needs an architecture fix — route these through the authenticated backend API instead of a direct privileged Supabase client — not a config change.
- **Order totals are entirely client-computed and written directly via that same service-role-requiring client.** `lib/orderService.ts:269-387` inserts `subtotal_amount`/`delivery_fee`/`total_amount` straight from client-computed values in `checkout.tsx`. Comments elsewhere claim "the backend derives the trusted amount from the DB row" — but that row's value was itself written by the client with no server recomputation, so the "trusted" amount is attacker-controlled at its source.
- **No session-to-userId binding on any Supabase-direct call — full IDOR.** `authService.ts:133-189`, `addressService.ts` (all CRUD), `orderService.ts:211-253,524-545` all take a bare `userId` string with zero check it matches the authenticated session's actual identity — the app doesn't even send the session token to these calls. Given the service-role key is (per above) necessarily reachable, anyone can query any other customer's full profile, saved addresses (with phone/lat-lng), and order history by ID alone.

### High (new)

- **Cart is never cleared after a successful order**, COD or paid — confirmed via grep, the only `clearCart()` call site is in the *failed/cancelled* payment branch. A customer can accidentally re-order the same items after checking out successfully.
- **No cart quantity ceiling and no stock check anywhere** — `CartContext.tsx` places no upper bound on quantity, and `orderService.ts`'s `createOrder` never validates against any inventory field before inserting.
- **`discount_amount` is hardcoded to `0` on every order** regardless of an applied coupon (`orderService.ts:375`) — the discount is folded into `total_amount` before being sent, but the dedicated column that finance/reporting would read is always wrong.
- **Coupon expiry is fetched but never enforced**, client or server side — an expired-but-still-`is_active` coupon (or a stale cached/tampered coupon object) remains applicable with zero re-validation at order-creation time.

### Medium (new)

- Widespread fire-and-forget `.catch(() => {})` patterns across cache writes and order-placed flags — individually low-risk, but no logging/telemetry means cache desyncs are invisible in production.
- Order-tracking screen runs a 5s fallback poll and a 2s driver-location poll unconditionally alongside the realtime channel, doubling backend load for no benefit when realtime is healthy.
- Order-confirmation screen finds the just-placed order by scanning the user's full order list client-side rather than fetching it directly — degrades to a silent empty state with no error/retry if the new order hasn't propagated into the (possibly cached) list yet.
- GSTIN input in checkout has no format validation before being persisted and later printed on the customer's own tax invoice.

### Low (new)

- 29+ more `console.log`/`warn`/`error` call sites beyond what's already known, no structured error tracking.
- `lib/supabase.ts` logs a "⚠️ CRITICAL" warning string to the on-device console in production if credentials are missing, rather than failing fast/reporting to a tracker.
- `invoiceService.ts` hardcodes literal placeholder GSTIN/phone strings (`'29XXXXXXXXXXXXX'`) as a silent fallback if business-info fetch fails — could ship on a real customer invoice.
- Biased shuffle (`Array.sort(() => Math.random() - 0.5)`) used for "suggested products" — harmless here, but a correctness smell if the pattern is copied elsewhere.

## Suggested order of attack

Given the sheer volume, this is not a "fix everything in one pass" list. Reasonable sequencing:

1. **Money-losing bug live right now**: unauthenticated refund endpoint (backend) — actively exposed to fraud today.
2. **Order-integrity bugs**: client-trusted checkout totals (backend + both frontend/customer-app checkout flows), rider order-hijack (backend ownership checks).
3. **Architecture-level fix**: customer app's service-role-key dependency — this is the biggest single fix (needs new backend endpoints + customer app changes) but unblocks fixing the IDOR issues under it too.
4. **Auth bypass bugs**: signup OTP-verification gap (backend), admin client-side login (admin panel).
5. Then work down through High → Medium → Low per area as time allows.

Let me know which of these you want tackled first — given the scope, I'd recommend starting with #1 and #2 since those are live financial/security exposure, not just latent risk.
