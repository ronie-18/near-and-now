## Session log — 2026-07-17, shopkeeper document verification feature

Deep-dive audit of the shopkeeper app's new "Upload Documents" page (added 2026-07-16, commit `4b48549`), followed by a rebuild and a second bug pass on that rebuild.

**Initial audit findings (all fixed by the rebuild below):** the mobile app's PATCH of a bulk `verification_documents` JSON blob was silently dropped (backend field not whitelisted, DB column never existed) — shopkeepers could "save" documents that never reached the server; the admin panel had no UI at all to view submitted documents before approving a store (see "Store approval is 'blind'" below, now fixed); the `store-documents` storage bucket was never provisioned, and the upload code called `getPublicUrl()` regardless of bucket privacy.

**Rebuild:** replaced the single-JSON-blob design with a proper per-document review workflow — new `store_verification_documents` table (one row per required doc: aadhaar/pan/trade/gst/fssai) instead of a JSONB blob, so admin can approve/reject each document individually with a reason. New backend endpoints for both the shopkeeper (`GET/POST .../store-owner/stores/:id/verification-documents(/:docType)`) and admin (`GET/PATCH .../api/admin/stores/:id/verification-documents(/:docType)`, `requireAdmin`-gated). Document uploads are proxied through the backend (multer, service-role Storage write) rather than direct-from-app to Supabase Storage — the shopkeeper app has no real Supabase Auth session (custom token system instead), so a client-side storage RLS policy couldn't actually be scoped to "only this shopkeeper's own store," and the anon key is baked into every install of the app; routing through the backend (which already authenticates via the existing token system and checks store ownership) avoids exposing every store's Aadhaar/PAN images to anyone who extracts that key. New admin "Review Documents" modal on `StoresPage.tsx`. Shopkeeper app shows a green tick (approved), red badge + rejection reason (rejected), or neutral (pending review) per document, plus PDF upload support (`expo-document-picker`) alongside the existing photo picker, with a 2MB/format disclaimer enforced both client-side and at the Supabase bucket level (`file_size_limit`/`allowed_mime_types`).

**Second bug pass on the rebuild, both fixed same session:**
- **Admin's rejection-reason text field leaked across documents.** `StoresPage.tsx`'s `DocumentReviewModal` kept the reject-reason textarea in a single shared `reason` state var for whichever document's panel was open; clicking "Reject" on a different document without confirming/cancelling the previous one left the old reason text in place — an admin could submit e.g. Aadhaar's rejection reason against the PAN card by mistake. Fixed by clearing `reason` whenever a different document's reject panel is opened.
- **Multer upload errors (e.g. oversized file) bypassed all JSON error handling.** No Express error-handling middleware existed anywhere in `backend/src/server.ts` (confirmed repo-wide); multer's `limits.fileSize` violation calls `next(err)` directly, before the route's own try/catch runs, so a file that slips past client-side size validation (e.g. `expo-image-picker`'s `asset.fileSize` is `undefined` on some devices) got Express's default non-JSON error response instead of a parseable one — the app could only show a generic "Failed to save document," not the real reason. Fixed by adding a catch-all Express error handler that returns proper JSON for `MulterError` (and any other unhandled error).

Not yet done: applying the migration was confirmed done by the user (2026-07-17); still pending is committing/pushing this work and on-device QA of the PDF picker and the 2MB rejection path.

## Session log — 2026-07-17, later pass — hardening the verification-documents rebuild

Continuation of the same-day rebuild above. Fixed a chain of missing-grant/RLS gaps discovered live while the user actually tested the feature (the exact "table was simply never grantable" bug class already documented elsewhere in this file, recurring here for a table created earlier the same day): `store_verification_documents` had RLS + a policy but no base `service_role` GRANT (backend calls failed with "permission denied for table"), then, once fixed, no `anon` GRANT either (the admin panel's direct-Supabase "Updated On" bulk read hit the identical error, silently, since it was wrapped in a non-fatal try/catch). Also added the table (plus `stores`, `products`, `customer_orders`, `store_orders`, `order_status_history`, `admin_notifications` — all of which already had realtime subscription code written against them in the apps but were never added to the `supabase_realtime` publication, so those subscriptions had been silently inert this whole time) to the realtime publication.

Feature additions on top of the rebuild: `stores.approved_at`/`approved_by` (who/when the store's final manual approval happened, cleared on revoke) and `store_verification_documents.approved_at`/`approved_by` (who/when a specific document was last approved — unlike `reviewed_at`/`reviewed_by`, deliberately *not* cleared by a later re-upload or rejection, so a document's approval history survives being resubmitted). Admin's Stores table gained an "Approved On" and "Updated On" column (live-updating via the newly-fixed realtime subscription + a 20s poll fallback, since it's unconfirmed whether Realtime's RLS check honors this app's custom `x-admin-token` header the same way normal REST requests do). Dropped `stores.verification_document`/`verification_number` — dead columns from the pre-rebuild single-document flow, confirmed unused anywhere except an unreachable allowlist entry. Raised the size limit from 2MB to 5MB; `file_size` changed from raw bytes to an already-formatted string ("340 KB"/"1.2 MB") computed once at upload time. Added a delete button (removes both the storage object and DB row, so a document can be re-uploaded from scratch) and a camera option (previously photo-library-only), plus a real format check (regex, not just presence) for Aadhaar/PAN/GST/FSSAI numbers — Trade License deliberately excluded, no fixed national format exists for it.

**Two significant bugs found on a final deep-dive pass, both fixed same session:**
- **Document preview silently vanished immediately after a successful save (shopkeeper app).** `app/upload-documents.tsx`'s `saveOne` replaced `serverDocs[key]` outright with the save endpoint's response — which is the raw DB row (`storage_path`, no signed `url`; signed URLs are only ever computed by the separate GET endpoints, generated fresh per-request since the bucket is private). The just-uploaded document's thumbnail blanked back to the empty "Upload X" placeholder right after a successful save, even though the upload had genuinely succeeded, until the screen was remounted and re-fetched real data. Same broken field also corrupted `handleSaveAll`'s completion message, which could say "upload the remaining documents" immediately after every document had just been successfully saved. Fixed by merging the response into the existing document object (keeping whatever `url` was already showing, or falling back to the just-picked local file's own path) instead of replacing it outright.
- **Document preview and Approve/Reject buttons vanished after any admin review action.** Identical root cause in `admin/src/pages/admin/StoresPage.tsx`'s `DocumentReviewModal`: the PATCH (approve/reject) endpoint's raw-row response has no `url` either, and `review()` replaced the local document object with it directly — both the thumbnail and the Approve/Reject buttons (gated on `doc.url` being truthy) disappeared the instant an admin reviewed *any* document, until the modal was closed and reopened. Same merge-instead-of-replace fix.

Both were confirmed by static review (re-reading the actual GET vs. PATCH/POST controller code side by side, not assumption) rather than live-device testing, so on-device/browser QA of both fixes is still outstanding.

**Feature request implemented same session: editing a document after approval suspends the store.** Previously, an already-approved shopkeeper could freely re-upload or delete any of the 5 documents with zero effect on `stores.is_approved` — by original design, per-document review was meant to stay fully independent from the store-level gate. Changed on request: `saveVerificationDocument`/`deleteVerificationDocument` now call a new `suspendStoreIfApproved()` helper that flips `is_approved` back to `false` and clears `approved_at`/`approved_by` (same fields a manual admin revoke clears) whenever the store *was* approved at the time of the edit — a no-op during normal first-time onboarding, since the store isn't approved yet. Both endpoints return a `storeSuspended` flag; the app shows an alert ("sent back for re-verification... contact admin") and redirects to `/pending-verification`.

**One more bug found immediately on deep-diving this new feature, fixed same session:** the redirect above would have been silently undone by client-side caching. `lib/appCache.ts` caches the store list (including `is_approved`) for 10 minutes; neither the save nor delete document calls ever invalidated it. So the instant a shopkeeper got redirected to `/pending-verification`, that screen's own `useStoreApprovalGate("require-pending")` would read the *stale, still-approved* cached store and immediately redirect them straight back into the app — completely undoing the suspension for up to 10 minutes. Fixed by calling the existing `clearStoreCache()` before navigating, forcing a real re-fetch that reflects the just-flipped `is_approved: false`. Not yet verified on-device.

## Pricing model / GST reference (confirmed by product owner, 2026-07-16)

Two independent, stacking GST layers — needed context for anything touching prices, checkout totals, or invoices:

1. **Per-item GST**, baked into each product's sellable price: `item_price = master_products.discounted_price + (discounted_price × master_products.gst_rate / 100)`. `discounted_price` is the pre-tax base. `is_loose = true` (loose/weighed products) → `gst_rate` is always 0. `is_loose = false` (packaged products) → follow the `gst_rate` column. Reference implementation: `frontend/src/utils/priceGst.ts`, mirrored in `admin/src/utils/priceGst.ts` and `frontend/src/services/supabase.ts:206-221`.
2. **A separate flat 5% GST on the whole bill at checkout**, mandated by GoI as a business-level tax (the rate can change if GoI changes it) — applied on top of the already-GST-inclusive item subtotal. This is intentional, not double taxation (see "Confirmed NOT a bug" under Website frontend below). Implemented in `frontend/src/utils/checkoutCalculations.ts`.

Any server-side price/total recomputation must replicate both layers or it will silently diverge from what the customer actually saw and agreed to pay — see the `placeCheckoutOrder` fix above, which initially missed layer 1.

## Multi-store order dispatch (`order_store_allocations` / `driver_order_offers`)

Found and fixed in a separate, targeted pass (not part of the original 7-area sweep above) while verifying the partial-acceptance → reallocation → driver-assignment flow end to end. All seven are now fixed. **Update 2026-07-23:** a new, distinct race was found in the acceptance step itself — see below.

### Reallocation flow (`shopkeeper.controller.ts`)

- ~~**Order permanently stalls if no nearby store stocks a reallocated item.**~~ — **FIXED 2026-07-16.** `reallocateMissingItems` only ever searched a fixed 4km radius and, on failure, marked the item `unavailable` with no path forward — the order stayed stuck in `store_accepted` forever since nothing re-checked whether it could still proceed. Now searches 4km, then widens to 8km (never further) for whatever's left; if still unresolved, flags the item for an admin-approved refund (new `admin_notifications` row, type `refund_required`, with the computed line-item amount and Razorpay payment ID) and lets the rest of the order proceed to dispatch instead of stalling. New admin panel action: "Refund ₹X" button on the flagged notification (`NotificationsPage.tsx`), backed by `POST /api/payment/resolve-item-refund/:notificationId` (`requireAdmin`-gated, validates against `customer_orders.refunded_amount` so a refund can never exceed what was actually paid). Migration: `20260719000000_customer_orders_refunded_amount.sql`.
- ~~**Order permanently stalls if the reallocated store never responds.**~~ — **FIXED 2026-07-16.** A newly-created `pending_acceptance` allocation had no timeout — if that shopkeeper never opened the app, the order sat waiting indefinitely. New `expireStaleAllocations(orderId)` runs opportunistically off `GET /api/tracking/orders/:orderId/full` (the endpoint the customer app already polls while an order is active) — any allocation still `pending_acceptance` after 5 minutes is auto-rejected and re-offered to the next nearest store via the same reallocation search. No new cron/infra needed.
- ~~**Race condition: two stores accepting near-simultaneously could double-broadcast to drivers.**~~ — **FIXED 2026-07-16.** The "is anything else still pending?" check was a plain read-then-write with no locking, so two concurrent acceptances could both conclude "I'm last" and both call `broadcastToNearbyDrivers` (mostly harmless in practice — the driver-offer upsert dedupes — but caused duplicate push notifications and a duplicate `order_status_history` row). Now atomic: new Postgres function `finalize_order_if_ready(p_order_id)` (migration `20260720000000_finalize_order_if_ready.sql`) locks the `customer_orders` row `FOR UPDATE` before deciding, mirroring the same pattern already used by `accept_driver_offer`.

- ~~**`acceptAllocation` has a check-then-act race with no atomic guard**~~ — **FIXED 2026-07-23** (found 2026-07-23, distinct from the store-vs-store double-broadcast race above — this one was in the single-store acceptance step itself). `shopkeeper.controller.ts:198-225` read the allocation, checked `alloc.status !== 'pending_acceptance'` in application code, then ran `update({status:'accepted', pickup_code:..., ...}).eq('id', allocationId)` with no `.eq('status', 'pending_acceptance')` guard on the update itself — two concurrent accept requests for the same allocation (a spotty connection causing the shopkeeper app's own retry logic to fire twice, or two devices logged into the same store account) could both pass the read-check before either wrote, both generate/store a pickup code, and both run the reallocation/finalize side effects twice. Fixed by moving the real guard onto the update's WHERE clause: `.eq('status', 'pending_acceptance')` is now part of the update itself, with `.select('id').maybeSingle()` to detect whether it actually matched a row. Only one of two concurrent requests can ever win that WHERE clause — the loser gets `null` back and returns a 409 ("Already responded") instead of generating a second pickup code or double-running the reallocation/finalize side effects. No new migration needed — this is an application-layer race (the check-then-act was in Node, not Postgres), unlike the store-vs-store race above which needed a DB-level row lock. `tsc --noEmit` clean; backend test suite unaffected (same 2 pre-existing, unrelated `payment.service.test.ts` failures as before).

### Driver broadcast flow (`shopkeeper.controller.ts` / rider app)

- ~~**Offline or suspended drivers could still accept a stale pending offer.**~~ — **FIXED 2026-07-16.** `acceptOffer` and the legacy `acceptOrder` only checked `is_approved`, never `is_online`/`status`, and neither did the `accept_driver_offer` Postgres function itself — so a driver who went offline or got suspended after receiving an offer could still accept it. Fixed at the real point of authority: `accept_driver_offer` now checks `is_approved`, `is_online`, and `status = 'active'` inside the same row-locked transaction (migration `20260721000000_accept_driver_offer_eligibility_check.sql`), returning `'driver_not_eligible'` if not. Both Node endpoints also fast-fail with a friendly 403 before calling the RPC.
- ~~**No cap on how many drivers get offered one order.**~~ — **FIXED 2026-07-16.** `broadcastToNearbyDrivers` offered every online/active driver within 10km with no limit — a busy area could push-notify 50+ drivers for one order. Now sorts candidates by distance (already computed, previously discarded) and caps the broadcast to the nearest 20 (`MAX_DRIVERS_PER_BROADCAST`).
- ~~**No staleness check on driver location.**~~ — **FIXED 2026-07-16.** Neither `broadcastToNearbyDrivers` nor `dispatchReadyOrdersToDriver` filtered on how recently `driver_locations` was updated — a driver marked `is_online=true` but whose app crashed/was killed hours ago would still be treated as "nearby" based on stale coordinates. Both now filter to locations updated within the last 5 minutes (`DRIVER_LOCATION_STALE_MS`).
- ~~**Up to 15s lag before an already-accepted order disappears from other drivers' lists.**~~ — **FIXED 2026-07-16 (rider app).** The rider app only had a Realtime subscription for new offer `INSERT`s, not `UPDATE`s — when another driver won an order, the loser's app didn't find out until its next 15s poll, showing a stale "available" card. `NAT_Near-Now_Rider-/app/(tabs)/home.tsx` now also subscribes to `UPDATE` events on `driver_order_offers` and removes the offer from local state the instant its status leaves `pending`.

## Customer app (`nearandnowcustomerapp`)

### Status of 6 previously-known issues

**Session model, clarified and implemented 2026-07-16:** intent is "stay logged in until explicit logout, or until 25 days of real inactivity — not 25/30 days since the original login." Two things were needed and both are now done:
- **Backend (`customerAuth.middleware.ts`):** `SESSION_TTL_MS` was a *fixed* 30-day window from `session_token_issued_at` (set once, at login) — a daily user would still get logged out on day 31 regardless of activity. Changed to 25 days and made it a genuine sliding window: every single authenticated request renews `session_token_issued_at` to the current time (fire-and-forget, doesn't block the response) — precise to the request, not throttled to a daily/coarser granularity, so the 25-day clock always counts from the customer's actual last activity down to the second. A customer who opens the app at least once every 25 days never gets logged out; 25 days of zero requests lets the token actually expire. (An earlier version of this fix throttled the renewal write to once/day to reduce DB load — reverted in favor of exact precision per an explicit product decision; worth reconsidering if this table's write volume ever becomes a real concern.)
- **App — the 401 handling was fixed, not just the token storage.** Previously `apiClient.ts` threw "Session expired" on a 401 but never cleared the stored token or updated the app's auth state — the customer stayed stuck in a broken "looks logged in, everything 401s" state instead of being dropped back to login. Added a `setSessionExpiredHandler` registration (`apiClient.ts` ↔ `AuthContext.tsx`) so any 401 from any API call — not just ones `AuthContext` itself makes — clears the session and flips `isAuthenticated` immediately, from wherever the user is in the app. Also added a global redirect in `app/_layout.tsx` (`AppShell`) watching the `isAuthenticated` true→false transition and sending the user to `/phone` — previously only `app/index.tsx`'s cold-start check redirected, so a session expiring mid-use (deep in checkout, tracking, etc.) had no path back to login at all.
- ~~**Bug found and fixed the same day: renewal only fired on specific ordering-related API calls, not on general app usage.**~~ — **FIXED.** Traced every code path and found the sliding-window renewal only runs inside `requireCustomer`, which only fires when the app hits a `requireCustomer`-gated backend route. Browsing (home/product/search) never does — `lib/productService.ts`/`lib/storeService.ts` talk to Supabase directly with the app's own client, bypassing the backend entirely. Worse, even the app's own "am I still logged in?" check on cold start (`AuthContext.restoreSession` → `getCurrentUserFromSession`) does the same — so simply *opening the app* didn't renew anything either; only placing/tracking an order, or touching addresses/email, happened to renew. Given the sheer volume, this is not a "fix everything in one pass" list. Reasonable sequencing, updated 2026-07-16 (later pass, then again after a QA-audit cross-check pass added 58 new findings from `QA_AUDIT.md`/`qa_audit_old.md`) to reflect what's already closed:
