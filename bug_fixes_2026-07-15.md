## The approval model (confirmed by deep dive)

The intended design â€” confirmed correct after tracing every route/trigger involved â€” is a **single admin gate**, not a repeated one:

- Admin approval is a one-time flag: `stores.is_approved` (boolean) for store owners, `delivery_partners.is_approved` (boolean, added 2026-07-16) for riders.
- Once set, it stays set. Nothing in the backend ever resets it back to pending/false except an explicit future admin action â€” confirmed by grepping every `UPDATE ... status`/`is_approved` write path in `backend/src/controllers` and `services/database.service.ts`.
- **`delivery_partners.is_approved` did not originally exist** â€” approval was inferred from the `status` enum (`active`/`inactive` = approved, `pending_verification`/`suspended`/`offboarded` = not), which conflated "has admin approved this person" with "what are they doing right now" (e.g. an admin-verified rider set to `inactive` would incorrectly read as unapproved). Added a real `is_approved` column, mirroring `stores.is_approved`, so it's now the single explicit source of truth for the gate â€” see item 6 below.
- After approval, **online/offline toggles and order accept/reject require no further admin involvement** â€” they only check session ownership, not admin state. Confirmed for:
  - Store owner: `acceptAllocation` / `rejectAllocation` / `completeAllocation` in `shopkeeper.controller.ts` â€” gated only by `requireShopkeeper` (session + store ownership), no admin check at all.
  - Delivery partner: `rejectOrder`, `markPickedUp`, `markDelivered`, `verifyPickupCode`, `verifyDeliveryOTP` in `deliveryPartner.controller.ts` â€” gated only by `requireRider` (session), no admin check.
- The **only** two places that should read the approval flag are the "go online" action and the "accept an order" action (the entry points into doing real work) â€” everything downstream of that trusts the session.
- Defense in depth on the rider side: a DB trigger (`20260428000001_decouple_is_online_from_status.sql`) forces `is_online = false` whenever `status <> 'active'` at the row level, independent of the API. The store side has no equivalent DB-level trigger for `is_active` â€” see "not yet fixed" below.
- **Rejection is intentionally not a distinct state.** Confirmed with the product owner: there is no "rejected" status, reason field, or dedicated UI to build, for either store owners or delivery partners. An application admin doesn't want to approve simply stays on `is_approved = false` / `status = pending_verification` forever, and the app shows the same "awaiting admin approval" screen indefinitely â€” indistinguishable from "not reviewed yet." `offboarded` (rider) is a separate concept reserved for partners who were previously active and are now being removed â€” it must never be used to deny a pending application.
- **Store owners must be fully locked out of the app until approved** â€” not just the online toggle. Browsing other tabs (Orders, Payouts, Inventory), viewing/adding products, and going online should all be blocked pre-approval; only Home (showing the pending-approval status) is accessible until admin approves. Confirmed and fixed in this session (previously only the online toggle was gated).

This session's fixes bring the actual code in line with that model (previously, delivery partners could skip the gate entirely, admin's UI couldn't even write the approval in the first place, and store owners could use the whole app while still pending).

## âś… FIXED (applied in this session)

### 1. [CRITICAL] Store approval silently did nothing

**File:** `admin/src/pages/admin/StoresPage.tsx`

The page used the plain anon Supabase client (`supabase`) instead of `getAdminClient()` (which attaches the `x-admin-token` header). The `stores` table's RLS policy only allows writes when `is_admin_authenticated()` passes, so `UPDATE is_approved` was silently blocked (0 rows affected, no error surfaced by PostgREST since no `.select()` was chained). The UI then optimistically flipped the badge anyway â€” **admins believed they were approving stores, but nothing was written to the database.**

**Fix:** Switched to `getAdminClient()` for both fetch and update, added `.select()` to the update so a blocked write now surfaces as a real error instead of a fake success.

### 2. [CRITICAL] Entire Delivery Partners admin page was non-functional (401s)

**File:** `admin/src/pages/admin/DeliveryPage.tsx`

Every fetch call (`fetchPartners`, `handleSave`, `handleDelete`, `handleToggleStatus`) hit `/api/delivery/partners*`, which is wrapped in `requireAdmin` on the backend â€” but none of the calls sent an `Authorization`/`x-admin-token` header. Every request would 401, and the failure was swallowed into a generic "Failed to load delivery partners" toast with no indication it was an auth bug. **The whole page â€” list, add, edit, delete, approve â€” was broken end-to-end.**

**Fix:** Added an `adminAuthHeaders()` helper (same `sessionStorage.getItem('adminToken')` pattern already used correctly in `OrderDetailPage.tsx`) and attached it to all four fetch calls.

### 3. [HIGH] No one-click Approve/Reject for delivery partners (parity gap vs. stores)

**File:** `admin/src/pages/admin/DeliveryPage.tsx`

Stores had a dedicated Approve/Revoke button; delivery partners had none â€” the only way to move a partner out of `pending_verification` was the generic Edit modal's status dropdown, and the "Status" pill was misleadingly clickable-looking for pending rows even though it always no-op'd with an error toast.

**Fix:** Added explicit **Approve** / **Reject** row actions that appear only for `pending_verification` partners. Approve sets `status: 'active'`. Reject keeps/resets `status: 'pending_verification'` â€” **not** `offboarded` (that status is reserved for previously-active partners being removed, and was the wrong choice in an earlier version of this fix; corrected after review â€” there is no distinct rejected state, see the approval-model note above).

### 4. [CRITICAL / SECURITY] Delivery partners could bypass admin approval entirely

**File:** `backend/src/controllers/deliveryPartner.controller.ts`

- `updateStatus` (go-online toggle) never checked `delivery_partners.status` â€” a rider still in `pending_verification` (or even `suspended`/`offboarded`) could set `is_online: true` directly via the API.
- `acceptOffer` (the live dispatch accept path the app actually uses) never re-checked the driver's status before accepting an order via the `accept_driver_offer` RPC.
- `acceptOrder` (a legacy/unused-by-current-app endpoint, still routed at `POST /delivery-partner/orders/:orderId/accept`) had the same gap â€” found during the follow-up deep dive.
- This was the mirror-image of the store-owner side, where `updateStoreStatus` **does** correctly gate on `is_approved` â€” delivery-partner approval was effectively decorative.

**Fix:** `updateStatus` (when going online), `acceptOffer`, and `acceptOrder` all now check the partner's approval flag and reject with `403` unless approved. (Initially implemented by checking `status === 'active'`; superseded by item 6 below, which adds an explicit `is_approved` column as the real source of truth.) This is a single lookup against the already-set approval flag â€” not a new admin process, and it doesn't fire again on every subsequent online/offline toggle or order action once the rider is approved (matches the "one-time gate" model above).

### 5. [HIGH] Unapproved store owners could use the whole app, not just skip the online toggle

**Files:** `backend/src/controllers/shopkeeper.controller.ts`, `near-now-store_owner/lib/useRequireStoreApproval.ts` (new), `app/(tabs)/previous-orders.tsx`, `app/(tabs)/payments.tsx`, `app/(tabs)/stock.tsx`

Confirmed as the intended design: a store owner who hasn't been approved yet should only be able to see Home (with its pending-approval status); Orders, Payouts, and Inventory should be completely inaccessible until admin approves. Previously `is_approved` was checked only inside the "go online" toggle â€” every other tab and its underlying API routes were wide open the moment signup completed, both from the app UI (no route guard) and directly against the API (no server-side check on `acceptAllocation`/`rejectAllocation`/`completeAllocation`/`getIncomingOrders`).

**Fix:**

- **Server-side:** `requireShopkeeper` (the middleware guarding all `/shopkeeper/*` routes â€” profile, orders, accept/reject/complete allocation) now checks that the owner has at least one approved store and returns `403 Your store is pending admin approval` otherwise. This is the same single-gate pattern as the rider fix: one check at the door, nothing downstream re-checks it.
- **Client-side:** added a `useRequireStoreApproval()` hook that the Orders, Payouts, and Inventory screens each call on mount â€” if the store isn't approved it redirects back to Home with an explanatory alert. This covers both tab-bar navigation and the Home screen's quick-action tiles (which navigate programmatically and would otherwise bypass a tab-bar-only guard).

### 6. [HIGH] `delivery_partners` had no `is_approved` column â€” approval was inferred from `status`, an ambiguous proxy

**Files:** `supabase/migrations/20260716000000_delivery_partners_is_approved.sql` (new), `backend/src/services/database.service.ts`, `backend/src/controllers/deliveryPartner.controller.ts`, `admin/src/pages/admin/DeliveryPage.tsx`

Raised directly: there was no explicit approval column for riders, unlike `stores.is_approved`. Approval was being read off the `status` enum (`active`/`inactive` = approved, everything else = not) â€” which is really an operational-lifecycle field, not an approval flag, and made the two concepts impossible to separate (e.g. no way to represent "admin-approved but currently suspended" distinctly from "never approved").

**Fix:**

- Added `delivery_partners.is_approved` (boolean, default `false`), backfilled `true` for existing `active`/`inactive` partners.
- `is_approved` is now the **sole** field the approval-gate checks read (`updateStatus`/go-online, `acceptOffer`, `acceptOrder` in `deliveryPartner.controller.ts` â€” previously checking `status === 'active'`, now checking `is_approved`), and the DB trigger that forces `is_online = false` for non-approved riders now keys off `is_approved` instead of `status`.
- `is_approved` is kept **auto-synced from `status`** inside `database.service.ts` (`isApprovedStatus()`: true for `active`/`inactive`, false for `pending_verification`/`suspended`/`offboarded`) whenever admin updates a partner's status â€” so the admin UI (Approve/Reject buttons, Edit form's status dropdown) needed **no changes**; it already only ever sets `status`, and the backend derives `is_approved` from that automatically.
- `status` itself is preserved as-is for operational lifecycle tracking (inactive/suspended/offboarded) â€” this is a decoupling, not a replacement, matching how `stores.is_approved` and `stores.is_active` are already two separate concerns.

## âš ď¸Ź NOT YET FIXED â€” recommended next (ranked by severity)

### Admin approval flow / cross-cutting

- **[HIGH] Store approval is "blind."** `StoresPage.tsx` never selects/renders `verification_document`, `verification_number`, or `owner_image_url` â€” columns that exist on `stores` and are shown for delivery partners. Admin can click Approve without ever seeing the shopkeeper's submitted ID.
- **[HIGH] No approval notification.** `notification.service.ts` only covers new-order events (`notifyShopkeeperNewOrder`, `notifyRiderNewOrder`). Approving a store or rider triggers no push/SMS/email â€” both mobile apps only find out via polling. (No rejection notification is needed â€” there's no rejection state; see the approval-model note above.)
- **[MEDIUM] No DB-level enforcement on the store side.** Unlike the rider's `is_online` trigger, nothing forces `stores.is_active` back to `false` if admin revokes `is_approved` on an already-online store â€” it stays online until the shopkeeper (or a future toggle attempt) touches it.
- **[MEDIUM] `OffersPage.tsx` (coupons) has the same missing-admin-header bug** as the old `DeliveryPage.tsx` â€” same root cause, not yet fixed, likely also broken end-to-end.
- **[MEDIUM] Admin login runs client-side.** `adminAuthService.ts` does `bcrypt.compare()` and reads the `admins` table (including `password_hash`) directly from the browser via the anon Supabase key, before any session exists. Should be a server-side login endpoint.
- **[MEDIUM] Delivery-partner signup has no server-side OTP gate.** `POST /delivery-partner/signup/complete` only validates `phone`/`name` in the body; nothing stops a client from calling it directly without having verified an OTP first (relies entirely on frontend flow order).
- **[LOW] `acceptOrder` (legacy) still has no ownership/offer check** â€” even with the approval gate added, an already-active rider could call it with an arbitrary `orderId` and get assigned to an order they were never offered. Confirmed the current rider app doesn't call this endpoint (it uses `/offers/:offerId/accept` exclusively), so it's dead code risk rather than an active exploit path â€” worth removing or locking down.
- **[LOW] Delivery OTP has no rate limiting** â€” 4-digit codes, brute-forceable, unlike admin login which has `is_account_locked()`.

### Store-owner app (`near-now-store_owner`)

- **[LOW/UX] `registration-success.tsx`** always redirects to Home after a flat 2.5s regardless of outcome, so there's no immediate "you're pending" moment right after signup.
- **[LOW]** `isActivated` field defaults to `true` when missing from the server response (fail-open), though it currently only affects a profile-page label, not access control.
- **[LOW]** `useRequireStoreApproval` (new, this session) fails open on network errors â€” a transient failure while checking approval lets the screen render rather than redirecting. Chosen deliberately so a flaky connection doesn't lock out an already-approved owner; means a not-yet-approved owner with a lucky-timed network blip could see the screen render for a moment before any retry. The server-side `requireShopkeeper` check is the real enforcement boundary â€” this client guard is UX-only.

### Rider app (`NAT_Near-Now_Rider-`)

- **[MEDIUM]** Approval detection is poll-only (30s interval, only while `home.tsx` is mounted with `driverStatus === "pending_verification"`) â€” no realtime subscription on the rider's own profile row, unlike order offers which do use realtime.
- **[MEDIUM] No `ErrorBoundary`** anywhere in the app â€” an unhandled render exception crashes the whole app with no fallback UI.
- **[LOW]** `isActivated` session field is dead code â€” set on signup/login but never read; real gating uses the live-polled `driverStatus`.
- Hardcoded admin support phone number baked into `home.tsx` rather than sourced from config.

### Customer app (`nearandnowcustomerapp`)

- **[HIGH] Logout doesn't clear the cart or push token.** On a shared/handed-down device, the next customer to log in inherits the previous customer's cart, and push notifications keep routing to the old push-token registration.
- **[HIGH] 401 handling is broken.** `apiClient.ts` throws "Session expired" on 401 but never clears stored auth state or signals `AuthContext` to log out â€” the UI stays stuck in a broken "authenticated" loop with no way back to login except manually opening settings.
- **[MEDIUM]** `trackingService.ts` treats any fetch failure during order tracking as "Order not found" instead of a retryable network error â€” alarming mid-delivery.
- **[MEDIUM]** Cold-start deep links from push notification taps can be dropped (no `router.isReady`/retry guard around `navigateFromPushData`).
- **[MEDIUM]** No push-token unregister on logout/uninstall â€” stale tokens can receive another account's order-status notifications.
- **[LOW]** `checkout.tsx`'s "Order Placed!" success overlay (~30 lines) is dead code â€” `setShowSuccess(true)` is never called; the app navigates straight to the confirmation screen instead.
- **[LATENT/risky if misconfigured]** `lib/supabase.ts` falls back to `EXPO_PUBLIC_SUPABASE_SERVICE_ROLE_KEY` if the non-public var is unset â€” documented as dev-only, but nothing prevents that var from accidentally being set in an EAS production build, which would bundle the service-role key into the shipped app.
